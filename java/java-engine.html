Write once Run anywhere

Java ist WORA (Write once Run anywhere). Java-Quellcode ist auf allen Betriebssystemen lauffähig. Vom Javac-Compiler generierter Bytecode (.class) kann auf jedem Betriebssystem ausgeführt werden. Bytecode ist verständlich, wenn jede JVM auf jedem Betriebssystem installiert ist.

JVM, JRE, JDK

JDK : Java-Entwicklungskit -Softwareentwicklungsumgebung, die zum Erstellen von Applets und Java-Anwendungen verwendet wird. JRE + Compiler + Debugger JRE : Java Runtime Environment-Software, die entwickelt wurde, um andere Software auszuführen. Um Java auszuführen , benötigen Sie JRE . Es enthält Klassenbibliotheken, Ladeklassen und JVM. JVM + Bibliotheken + andere Komponenten (zum Ausführen von Applets / Java-Anwendungen) JVM : Virtuelle Maschine, auf der Java-Bytecode ausgeführt wird macht java portabel kommt mit JIT (Just-In-Time)-Compiler, der Java-Quellcode in Low-Level-Maschinensprache konvertiert.

Die Java-JIT-Kompilierung

Als Nächstes ist es Zeit für den Interpreter (Dolmetscher).
Just-In-Time (JIT)-Kompilierung, auch als dynamische Kompilierung bekannt . Das Programm wird während der Laufzeit in nativen Code kompiliert, um die Leistung zu verbessern. Bei der JIT-Kompilierung übersetzt bytecode(.class) in Maschinencodeanweisungen von laufenden Maschinen Der resultierende Maschinencode ist für die CPU-Architektur der laufenden Maschine optimiert Die Kompilierung erfolgt zur Laufzeit des Programms (im Gegensatz zur Ausführung) Da die Kompilierung zur Laufzeit erfolgt, hat der JIT-Compiler Zugriff auf dynamische Laufzeitinformationen, wodurch er bessere Optimierungen vornehmen kann Die Hoffnung ist, dass die Effizienz der Ausführung von Maschinencode die Ineffizienz der Neukompilierung von Programmen bei jeder Ausführung überwindet. Optimierungen JVM führt Bytecode aus und zählt, wie oft die Funktion ausgeführt wird . Wenn die Anzahl das vordefinierte Limit überschreitet, kompiliert JIT Code in Maschinensprache, der direkt vom Prozessor ausgeführt werden kann. Beim nächsten Mal wird die Funktion berechnet, der kompilierte Code wird im Gegensatz zur normalen Interpretation erneut ausgeführt --> schnellere Ausführung

Heap- und Stack-Speicher

Um die Anwendung optimal auszuführen, teilt JVM den Speicher in Stack & Heap-Speicher Stack auf : wird für die Ausführung des Threads verwendet Primitive Werte, die für eine Methode spezifisch sind Verweise auf Objekte, die sich in einem Heap befinden, auf die von der Methode verwiesen wird. thread-safe : eahc Thread wird in einem eigenen Stack betrieben Heap : tatsächliche Objekte dynamische Speicherzuweisung für Java-Objekte und JRE-Klassen zur Laufzeit String-Pool nicht Thread-sicher : muss durch ordnungsgemäß synchronisierenden Code geschützt werden Neue Objekte werden immer im Heap-Space erstellt und die Referenzen auf diese Objekte werden im Stack-Speicher abgelegt. Wenn der Haufen voll ist, wird Müll gesammelt. So werden nicht mehr benötigte Gegenstände geräumt und so Platz für neue Objekte geschaffen. Haufen Wird beim Start von JVM erstellt und kann während der Ausführung der Anwendung größer/kleiner werden Wenn der Haufen voll ist, wird Müll gesammelt Neue Objekte, die im Heap-Space erstellt wurden, verweisen auf Objekte, die im Stack-Speicher gespeichert sind Objekte haben globalen Zugriff und können von überall in der Anwendung aufgerufen werden, nicht Thread-sicher Zerlegt in 3 Teile Junge Generation : Alle neuen Objekte werden zugewiesen und gealtert. Old / Tenured Generation : lange überlebende Objekte gespeichert. Objekt gespeichert in Young Gen, Schwellenwert für sein Alter festgelegt. wenn die Schwelle erreicht ist. Objekt verschoben auf alte Gen Permanente Generierung : JVM-Metadaten für Laufzeitklassen und Anwendungsmethoden. Wenn der Haufen voll wird, wird Müll gesammelt Stapel in LIFO-Reihenfolge referenziert Wenn eine neue Methode aufgerufen wird, wird ein neuer Block oben auf dem Stapel erstellt, der für diese Methode spezifische Werte enthält (z. B. primitive Variablen und Verweise auf Objekte). Methode beendet Ausführung -> entsprechender Stapelrahmen wird geleert, Ablauf geht zurück zum Aufruf der Methode. Speicherplatz für die nächste Methode verfügbar

  1. Programm ausführen -> Laufzeitklassen in Heapspace laden
  2. main()-Methode in Zeile 1 gefunden, erstellt zur Laufzeit Stack-Mem, das vom Thread der main()-Methode verwendet werden soll. Immer wenn ein neues Objekt erstellt wird,
  3. im Heap-Speicher erstelltes Objekt
  4. Stack-Speicher enthält die Referenz dafür.
  5. wenn neue Methode aufgerufen wird -> neuer Block auf Stack erstellt (LIFO)
  6. String, Referenz im Stapelspeicher -> Verweise auf den String-Pool im Heap-Space
  7. Wenn die Methode (foo()) beendet wird, wird der Stapel frei (der für die Funktion erstellte Stapelspeicher wird zerstört)
  8. gehe zum nächsten Programm unten (main())
